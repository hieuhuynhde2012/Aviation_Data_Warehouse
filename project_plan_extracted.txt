5. Airflow DAG và luồng xử lý
Airflow sẽ thể hiện kỹ năng orchestration: dependency, retry, timeout, scheduling, logging, backfill và 
failure handling. DAG chính nên chạy daily, còn DAG phụ dùng cho backfill theo khoảng ngày/tháng.
start
  -> generate_or_download_sources
  -> upload_raw_files_to_minio
  -> validate_raw_schema_and_row_count
  -> load_raw_csv_to_postgres
  -> run_dbt_staging
  -> run_dbt_intermediate
  -> run_dbt_marts
  -> run_dbt_tests
  -> update_pipeline_metadata
  -> refresh_superset_datasets
  -> send_pipeline_summary
end

5.1. Task breakdown
Task Output Failure handling
generate_or_download_sources CSV files trong /data/input Retry 2 lần; fail nếu thiếu file bắt buộc.
upload_raw_files_to_minio Object trong MinIO raw bucket Checksum sau upload; archive duplicate.
validate_raw_schema_and_row_count Validation report Fail nếu thiếu column, row_count = 0, sai 
encoding.
load_raw_csv_to_postgres raw_* tables Transaction rollback nếu load lỗi.
run_dbt_staging/intermediate/marts stg/int/mart tables dbt exit code != 0 thì fail DAG.
run_dbt_tests test results Fail nếu unique/not_null/relationship test 
không pass.
update_pipeline_metadata pipeline_run_log Ghi cả success/failure để trace.

6.1. Tests nên có
 not_null: booking_id, flight_id, airport_code, flight_date, ticket_price.
 unique: booking_id, payment_id, airport_code trong dimension.
 accepted_values: booking_status, payment_status, airport_type.
 relationships: origin_airport/destination_airport phải tồn tại trong dim_airport nếu code match được.
 custom test: ticket_price >= 0, delay_minutes không âm khi delay flag true, row_count mart không vượt bất 
thường so với source.
6.2. Incremental strategy
Với bảng lớn như fact_booking và mart_booking_trend_daily, nên dùng incremental model theo partition 
date. Khi rerun một ngày cụ thể, chỉ rebuild partition đó thay vì rebuild toàn bộ warehouse.
